## Airflow.cfg
- 위치: Airflow가 설치된 디렉토리 밑, dags_folder 키에 저장됨
- 새로 생성한 Dag의 스캔 주기를 결정하는 키 - dag_dir_list_interval(default 300초)
- Airflow를 API 형태로 외부에서 조작하고 싶다면 
    + api 섹션의 auth_backend를 airflow.api.auth.backend.basic_auth로 변경
- Variable에서 변수의 값이 encrypted가 되려면 변수의 이름에 아래 단어들이 포함되어야 함
    + password, secret, passwd, authorization, api_key, apikey, access_token
- 환경 설정 파일을 수정했다면 실제로 반영하기 위해 해야 하는 일
    + docker 재실행
    + sudo systemctl restart airflow-webserver
    + sudo systemctl restart airflow-scheduler
- fernet_key: Metadata DB를 암호화하는데 사용되는 키

## Airflow와 타임존
- airflow.cfg에는 두 종류의 타임존 관련 키가 존재
    + default_timezone
    + default_ui_timezone
- start_date, end_date schedule
    + default_timezone에 지정된 타임존을 따름
- execution_date와 로그 시간
    + 항상 UTC를 따름
    + 즉 execution_date를 사용할 때는 타임존을 고려해서 변환 후 사용필요
- 가장 좋은 방법은 UCT를 일관되게 사용하는 것으로 보임

## dags폴더에서 코딩시 작성한다면 주의할 정
- Airflow는 dags 폴더를 주기적으로 스캔함 - dag_dir_list_interval = 300
- 이때 DAG 모듈이 들어 있는 모든 파일들의 메인 함수가 실행이 됨
    + 이 경우 본의 아니게 개발 중인 테스트 코드도 실행될 수 있음
    ```
    from airflow import DAG
    cur.execute("DELETE FROM ...")
    ```


## Open Weathermap API 
### 소개
- 위도/경도를 기반으로 그 지역의 기후 정보를 알려주는 서비스
- 무료 계정으로 api key를 받아서 호출시에 사용
- https://openweathermap.org/price

### 만들려는 DAG
- 8일 서울의 낮/최소/최대 온도 읽기
- API Key를 open_weather_api_key라는 Variable로 저장
- 서울의 위도와 경도를 찾을 것
- One-Call API를 사용: https://openweathermap.org/api/one-call-api
    + 앞서 API KEY와 서울의 위도/경도를 사용해서 위의 API를 requests 모듈을 사용해서 호출
    + 응답 결과에서 온도 정보(평균/최소/최대)만 앞의 8일을 대상으로 출력할 것
        - 날짜, 낮 온도(day), 최소 온도(min), 최대 온도(max)

### DAG구현
- Open Weathermap의 one call API를 사용해서 weather_forecast 테이블에 저장
    + created_date는 레코드 생성시간(default)으로 자동 채워지는 필드이다 
- One-call API는 결과를 JSON형태로 리턴해줌
    + requests.get(linek).json()을 사용
    + daily 필드는 리스트이며 각 레코드가 하나의 날짜에 해당, 
    + 날짜 정보는 dt 필드에 존재, epoch라고 1970년 1월 1일 이후 밀리세컨드로 시간을 표시
        - 읽을 수 있는 날짜로 변경 - datetime.fromtimestamp(d["dt"]).strftime("%Y-%m-%d") 
- 두 가지 방식의 Full refresh 구현 방식
    + insert into, COPY 
- autocommit의 값을 False로 진행

### Open Weather API 호출 응답 보기
- daily라는 리스트에 앞으로 7일간의 온도 정보가 들어옴
    + dt필드가 날짜를 나타냄
    + temp필드가 온도 정보를 나타냄
        - day, min, max, night, eve, morn
    


## Primary key Uniqueness보장하기
### Primary key Uniqueness란?
- 테이블에서 하나의 레코드를 유일하게 지칭할 수 있는 필드
    + 하나의 필드가 일반적이지만 다수의 필드를 사용할 수도 있음
    + CREATE TABLE 사용시 지정
- 관계형 데이터베이스 시스템이 Primary key값이 중복 존재하는 것을 막아줌
    + User테이블의 Email필드, Products테이블에서 product_id 필드
- SQL query를 이용한 Primary key 지정하는 방법
```sql
CREATE TABLE products(
    product_id INT PRIMARY KEY,
    name VARCHAR(50),
    price decimal(7, 2)
);

CREATE TABLE orders(
    order_id INT,
    product_id INT,
    PRIMARY KEY (order_id, product_id)
    FOREIGN KEY (product_id) REFERENCES products (product_id)
);
```

### 빅데이터 기반 데이터 웨어하우스들은 Primary Key를 지켜주지 않음
- Primary key를 기준으로 유일성 보장을 해주지 않음 
    + 이를 보장해주는 것이 데이터 인력의 책임
- Primary key 유일성을 보장해주지 않는 이유는?
    + 보장하는데 메모리와 시간이 더 소요되기 때문에 대용량 데이터의 적재에 걸림돌이 됨

### Primary key 유지 방법
- 날씨 정보의 경우 최근 정보가 더 신뢰할 수 있음
- 어느 정보가 최근 정보인지 created_date필드에 기록하고 활용하는 방식
- 즉, Primary key가 같은 데이터를 SQL문법 ROW_NUMBER를 활용해 최근 정보를 선택
```sql
CREATE TABLE yeojun.weather_forecast(
    date date primary key,
    created_date timestamp default GETDATE(),
    ROW_NUMBER() OVER(partition by date order by created_date DESC) seq
);
```
- 임시 테이블(스테이징 테이블)을 만들고 현재 모든 레코드를 복사
- 새로 데이터소스에서 읽어들인 레코드들을 복사(중복 존재 가능성 있음)
- 중복을 걸러주는 SQL을 바탕으로 최종 데이터를 원본 테이블로 복사

### Upsert란?
- Primary Key를 기준으로 존재하는 레코드라면 새 정보로 수정
- 존재하지 않는 레코드라면 새 레코드로 적재
- 보통 데이터 웨어하우스마다 UPSERT를 효율적으로 해주는 문법을 지원해줌








## Backfill과 Airflow
### 데이터 갱신 방식의 특징 및 고려사항
- Full Refresh는 문제가 생겨도 재실행을 통해 해결할 수 있음
- Backfill은 Incremental Update시에만 의미가 있음
- Incremental Update는 효율성이 더 좋을 수 있지만 운영/유지보수의 난이도가 올라감
    + 실수등으로 데이터가 빠지는 일이 생길 수 있음
    + 과거 데이터를 다시 다 읽어와야하는 경우 모두 재실행을 해주어야함

### Backfill
- Backfill의 정의
    + 실패한 데이터 파이프라인을 재실행 혹은 읽어온 데이터들의 문제로 다시 읽어와야하는 경우를 의미
- Backfill의 해결은 Incremental Update에서 복잡해짐
    + Full refresh에서는 간단, 그냥 다시 실행하면 됨
- 실패한 데이터 파이프라인의 재실행이 얼마나 용이한 구조인가?
    + 해당 역할은 데이터 엔지니어의 삶에 직접적인 영향을 끼친다.
    + 이 역할이 잘 디자인된 플랫폼이 Airflow이다.

### Daily DAG
- 지금 시간을 기준으로 어제 날짜를 계산하고 그 날짜에 해당하는 데이터를 읽어옴
```python
from datetime import datetime, timedelta

# 지금 시간 기준으로 어제 날짜를 계산
y = datetime.now() - timedelta(1)
yesterday = datetime.strftime(y, '%Y-%m-%d')

# yesterday에 해당하는 데이터를 소스에서 읽어옴
# 예를 들어 프로덕션 DB의 특정 테이블에서 읽어온다면
sql = f"SELECT * FROM table WHERE DATE(ts)="{yesterday}"
```
- 지난 1년치 데이터를 Backfill 해야한다면?
    + 기존 ETL 코드를 조금 수정해서 지난 1년치 데이터에 대한 DAG를 실행한다
    + 실수하기 쉽고 수정하는데 시간이 많이 소요될 것이다

### Backfill이 용이하도록 ETL 구현하는 방법
- 시스템적으로 구현하기
    + 날짜별로 Backfill 결과를 기록하고 성공 여부 기록: 추후에 결과를 쉽게 확인
    + 날짜를 시스템에서 ETL의 인자로 제공
    + 데이터 엔지니어는 읽어와야하는 데이터의 날짜를 계산하지 않고 시스템이 지정해준 날짜를 사용
- Airflow의 접근방식
    + ETL 별로 실행날짜의 결과를 메타데이터 데이터베이스에 기록
    + 모든 DAG 실행에는 "execution_date"이 지정되어 있음
        - execution_date으로 채워야하는 날짜와 시간이 넘어옴
    + 이를 바탕으로 데이터를 갱신하도록 코드를 작성
    + 즉 Backfill의 쉬워짐

### start_date와 execution_date 이해하기
- 2020-08-10 02:00:00로 start date로 설정된 daily job이 있다.
    + catch up은 default값인 True
- 지금 시간이 2020-08-13 20:00:00이고 처음으로 Job이 활성화 되었다.
- 이 경우 job은 몇번 실행되었을까? 총 3회
    + start_date 2020-08-11 02 / execution_date 2020-08-10 02
    + start_date 2020-08-12 02 / execution_date 2020-08-11 02
    + start_date 2020-08-13 02 / execution_date 2020-08-12 02

### Backfill과 관련된 Airflow변수들
- start_data 
    + DAG가 처음 실행되는 날짜가 아니라 DAG가 처음 읽어와야하는 데이터의 날짜/시간
    + 실제 첫 실행날짜는 start_date + DAG의 실행주기(Daily일 경우 다음날)
- execution_date
    + DAG가 읽어와야하는 데이터의 날짜와 시간
- catchup
    + DAG의 첫 시점이 start_date보다 미래라면 그 사이 DAG에 대한 실행 여부를 결정하는 파라미터
    + True가 디폴트 값이고 이 경우 실행되지 않은 것들을 모두 따라잡으려고 함
    + False가 되면 실행되지 않은 것들을 무시함
- end_date
    + 보통 필요하지 않은 값
    + Backfill을 날짜 범위에 대해 하는 경우에만 필요
