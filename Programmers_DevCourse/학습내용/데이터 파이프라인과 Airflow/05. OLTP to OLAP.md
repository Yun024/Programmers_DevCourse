### AWS 관련 권한 설정
- Airflow DAG에서 S3 접근(쓰기 권한)
    + IAM User를 만들고 S3 버킷에 대한 읽기/쓰기 권한 설정하고 access key와 secret key를 사용
- Swnoflake가 S3 접근(읽기 권한)
    + Snowflake에 S3를 접근할 수 있는 역할(Role)를 만들고 이를 Snowflake에 지정

### AWS S3 접근(Connections)
- IAM을 사용해 별도의 사용자를 만들고 새로 만든 Bucket만을 읽고 쓸 수 있는 권한을 제공
    + AmazonS3FullAccess가 아닌 Create policy를 사용
- Access Key ID와 Secret Access Key를 사용 - 주기적으로 변경하여 해킹의 위험을 최소화
1. Add users - airflow-s3-access 사용자 생성
2. Create policy를 이용한 Policy 직접 추가
3. Custom Policy의 내용 설정
4. IAM -> Users -> airflow-s3-access 선택
    + Security credentials - Create access key
5. S3 Connection 설정
    + Conn Id: aws_conn_id
    + Conn Type: `S3` or `Amazon Web Service` or `Generic`
    + Extra: {"region_name":"us-west-2"}

## Backfill 실행해보기
### 2023년 1월달 데이터를 다시 읽어와야 한다면
- CLI에서 하루씩 31번 실행
    + airflow dags test MySQL_to_Snowflake_v2 2023-01-01 부터
    + airflow dags test MySQL_to_Snowflake_v2 2023-01-31 까지
- 한번에 여러 날짜를 동시에 실행 가능한가?
    + 구현 방법에 따라 한번에 하나씩 실행하는 것이 안전할 수 있음
    + 이를 제어해주는 DAG 파라미터가 `max_active_runs`
- 여러 날짜 Backfill을 커맨드라인에서 실행하는 방법
    + airflow dags backfill MySQL_to_Snowflake_v2 -s 2023-01-01 -e 2023-01-31
    + catchUp가 True로 설정되어 있어야 함
    + execution_date를 사용해서 Incremental update가 구현되어 있어야 함
    + start_date부터 시작, end_date은 포함하지 않음(전날 데이터까지 Load)
    + 실행 순서는 날짜/시간순은 아니고 랜덤
        - 날짜순으로 하고 싶으면 Dag default_args의 depends_on_past를 True로 설정
    
### DAG를 backfill에 대비하여 설정하기
- 모든 DAG가 backfill을 필요로 하지 않음
    + Full refresh를 한다면 backfill은 의미가 없음
- backfill은 일별 혹은 시간별로 업데이트하는 경우를 의미함
    + 마지막 업데이트 시간 기준 backfill을 하는 경우(Data Warehouse 테이블에 기록된 시간 기준)
        - 이런 경우에도 execution_date를 이용한 backfill은 필요하지 않음
- 데이터의 크기가 굉장히 커지면 backfill 기능을 구현해 두는 것이 필수
    + 에러 상황에 대하여 airflow가 큰 도움이 됨 
    + 데이터 소스의 도움 없이는 불가능
- 어떻게 backfill을 구현할 것인가
    + 제일 중요한 것은 데이터 소스가 backfill 방식을 지원해야 함
    + execution_date 를 사용해서 업데이트할 데이터 결정
    + catchup 필드를 True로 설정
    + start_date/end_date 를 backfill하려는 날짜로 지정
    + DAG 구현이 execution_date를 고려해야 하는 것이고 idempotent해야함
