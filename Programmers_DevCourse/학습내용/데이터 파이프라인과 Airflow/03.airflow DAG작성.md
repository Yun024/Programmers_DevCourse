## PythonOperator
아래는 PythonOperator를 사용하는 예제코드이다.
- cxt는 사용자가 지정한 변수, "Airflow에서 Task가 실행될 때 Task의의 정보를 담은 딕셔너리" 역할
- cxt['params']는 params={'table':'table1'}를 불러온다.
- cxt['params']['table']는 'table1'을 불러온다. 즉 params['table']의 키값
```python
from airflow.operators.python import PythonOperator # 불러오기
def python_func(**cxt):             # 아래 python_callable에서 사용될 함수 정의
    table=cxt["params"]["table"]    # 필요한 함수 기능 
    schema=cxt["params"]["schema"]

    ex_date = cxt["execution_date"]
    # 필요한 것에 대해 작성

load_nps = PythonOperator(
    dag=dag,                        # dag지정
    task_id="taks_id",              # task_id 무엇인지
    python_callable = python_func,  # task가 실행될 때 실행되야 할 함수 지정
    params={                        # 함수에 필요한 인자
        'table':'delighted_nps',
        'schema':'raw_data'
    }
)
```

### Hello World 예제 프로그램
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

dag = DAG(
    dag_id = 'HelloWorld',
    start_date = datetime(2022,5,5),
    catchup=False,
    tags=['example'],
    schedule = '0 2 * * *')

def print_hello():
    print("hello!")
    return "hello!"

def print_goodbye():
    print("goodbye!")
    return "goodbye!"

print_hello = PythonOperator(
    task_id = 'print_hello',
    #python_callable param points to the function you want to run 
    python_callable = print_hello,
    #dag param points to the DAG that this task is a part of
    dag = dag)

print_goodbye = PythonOperator(
    task_id = 'print_goodbye',
    python_callable = print_goodbye,
    dag = dag)

#Assign the order of the tasks in our DAG
print_hello >> print_goodbye


## Airflow Decorators
from airflow import DAG
from airflow.decorators import task
from datetime import datetime

@task
def print_hello():
    print("hello!")
    return "hello!"

@task
def print_goodbye():
    print("goodbye!")
    return "goodbye!"

with DAG(
    dag_id = 'HelloWorld_v2',
    start_date = datetime(2022,5,5),
    catchup=False,
    tags=['example'],
    schedule = '0 2 * * *'
) as dag:

    # Assign the tasks to the DAG in order
    print_hello() >> print_goodbye()

똑같은 기능을 수행하지만 프로그래밍이 단순해짐
Airflow Decorators를 사용하면 함수를 정의하는 동시에 그 자체가 task가 됨

## 중요한 DAG 파라미터
with DAG(
    # DAG에 직접 지정해주는 파라미터들
)
- max_active_runs: 동시에 실행할 수 있는 DAG의 수, cpu의 총합이 upper-bound
- max_active_tasks: 동시에 실행할 수 있는 Task의 수, cpu의 총합이 upper-bound
- catchup: start_date가 현재보다 과거일 때 밀린 날짜에 대한 DAG를 채워준다. 
    + Incremental Update하는 DAG에서 중요하다: 변경 사항이나 추가된 부분만 업데이트
    + full_refresh: 기존 데이터의 변경 여부와 관계없이 모든 데이터를 다시 읽어 갱신 및 덮어쓰기 


- DAG parameters vs Task parameters의 차이점

DAG Parameters vs Task Parameters
특징	DAG Parameters	Task Parameters
범위	DAG 전체 (전역 설정)	특정 Task에만 국한됨
설정 위치	DAG 정의 시 설정 (DAG 클래스의 매개변수)	Task 정의 시 설정 (Operator의 매개변수)
목적	DAG 전반적인 설정 및 공통 변수 관리	개별 Task의 동작 제어 및 실행 세부사항 설정
접근 방식	모든 Task에서 접근 가능	해당 Task 내에서만 접근 가능
예시	schedule_interval, default_args, catchup 등	bash_command, retries, op_args, op_kwargs 등

### Xcom이란
하나의 태스크로 구성할 경우 내가 직접 정의해서 데이터를 주고받지만 여러 태스크가 있을 경우 어떻게 데이터를 주고받을까?
- 태스크(Operator)들간에 데이터를 주고 받기 위한 방식
- 보통 한 Operator의 리턴값을 다른 Operator에서 읽어가는 형태가 됨(task_id사용)
- 이 값들은 Airflow 메타 데이터 DB에 저장이 되기에 큰 데이터를 주고받는데 사용할 수 없음
    + 보통 큰 데이터는 S3등에 로드하고 그 위치를 넘기는 것이 일반적

### DAG에서 task를 어느 정도로 분리하는 것이 좋을까?
- task를 많이 만들면 전체 DAG이 실행되는데 오래 걸리고 스케줄러에 부하가 감
- task를 너무 적게 만들면 모듈화가 안되고 실패시 재실행 시간이 오래 걸림
- 오래 걸리는 DAG는 재실행이 쉽도록 다수의 task로 나누는 것이 좋음

### Airflow의 Variable 관리 vs 코드관리
- 장점: 코드 푸시의 필요성이 없음
- 단점: 관리나 테스트가 안되어서 사고로 이어질 가능성이 있음
