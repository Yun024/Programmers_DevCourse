# Docker Voloume
## Docker Volumne의 필요성
- 특정 소프트웨어가 Docker Container를 통해 일회성으로 동작하는 것이 아님
    + 데이터베이스가 Docker Container안에서 동작 => 데이터가 영구적으로 보관되어야 함
- 이를 보장하는 기능이 Docker Volumes => Data Persistence
- Docker Container내의 가상 파일 시스템과 호스트 시스템의 파일 시스템을 맵핑
    + `/home/yeojun024/logs` 호스트 시스템의 파일 시스템
    + `/var/lib/airflow/logs` Docker Container의 가상 파일 시스템
    + 둘을 맵핑함으로써 Docker Container가 중단되더라도 모든 Airflow logs는 기록이 남게 됨

## Docker Volume 예
- Container와 Host 시스템이 특정 폴더를 공유
    + Container가 사라지더라도 데이터는 남게 하는 것
    + Container로 MySQL을 실행하는 경우 데이터가 저장되는 공간을 Docker Volume으로 설정

## Docker Volume 정의
- 호스트 파일 시스템 폴더를 Docker Container 파일 시스템의 폴더로 마운트하는 것
    + 호스트쪽에서 내용을 바꾸면 바로 Docker Container쪽에도 반영됨
        - 반대도 가능하지만 Volume의 속성을 잡기 나름(예를 들면 읽기 전용)
    + 이를 통해 Docker Container의 상태와 관계없이 데이터를 영구적으로 보관
- 파일 시스템에서 마운트란?
    + 원래 마운트란 디스크와 같은 물리적인 장치를 파일 시스템의 특정 위치에 연결해주는 것
    + 윈도우에서 USB를 꼽으면 내 컴퓨터에 새로운 :D, :E와 같이 새로운 폴더가 생기는 것 

## Docker Volume 타입
1. Host Volumes: docker run -v를 실행할 때 페어로 지정
    + docker run -v /home/yeojun/logs:/var/lib/airflow/logs
    + ":"앞이 호스트 파일 시스템, 뒤가 컨테이너 파일 시스템의 경로 
2. Anonymous Volumes: docker run -v를 실행할 때 컨테이너 경로만 지정
    + docker run -v /var/lib/mysql/data
    + 이 방식이 Dockerfile에서 사용되는 방식으로 호스트쪽에 액세스되지는 않지만 재시작해도 유지
3. Named Volumes: docker run -v를 실행할 때 이름과 컨테이너 경로를 지정
    + docker run -v name:/var/lib/mysq/data
    + 가장 선호되는 방식. 하나의 Volume을 다수의 컨테이너에서 공유하는 것도 가능하게 해줌
※ Volume을 Readonly로 지정하고 싶다면 - name:/var/lib/mysql/data:ro

## Docker Volume 사용법
1. Dockerfile
    - VOLUME 명령을 통해 anonymous volume만 지정 가능
2. docker-compose
    - Host Volume이나 Named Volume을 사용하는 것이 일반적

# Docker-Compose
## Docker-Comopse란?
- 다수의 Container로 소프트웨어가 구성되는 경우 사용할 수 있는 툴 + 환경설정파일
    + docker-compose.yml로 설정
    + 다양한 테스트등도 수행가능
    + 다양한 버전을 만드는 것도 일반적(dev, test, prod 등등)
- 개별 Container를 따로 관리하는 것보다 훨씬 더 생산성이 높음
- 환경설정 파일의 이름은 docker-compose.yml 이나 docker-compose.yaml 

- Docker Desktop의 일부로 설치가 됨

- Docker Compose v.1.27+ 부터 v2와 v3가 합쳐짐
- 네트워크 설정을 자동화하고 관리하기 쉽게 만들어 개발 및 배포 생산성을 크게 향상시킴

## docker-compose 명령어
- 사용법 자체는 간단: 소프트웨어를 구성하는 모든 컨테이너에게 적용됨
- docker compose up
    + build => create => start
    + docker compose build
        - build 키로 지정된 것들 대상
    + docker compose create
    + docker compose start
- docker compose down
    + stop => remove
    + docker compose stop
    + docker compose rm
- docker compose ps
- docker compose ls
    + docker-compose를 그룹별로 보여줌
- docker compose push
    + docker hub으로 이미지들을 푸시함
- docker compose pull
    + docker hub에서 이미지들을 읽어오려고 함
- docker images
    + 각 개별 이미지 앞에 폴더 이름을 prefix로 붙임(docker hub에서 읽어온 것 제외)
- docker compose images
    + 컨테이너에 의해 실행되고 있는 이미지들만 보여줌
- docker compose version(docker-compose --version)
    + docker Engine을 실행하고 터미널에서 버전 확인
    

## docker-compose.yml 작성
- 다양한 버전이 존재하지만 v2와 v3가 합쳐진 버전 사용 예정
- services: 다음으로 프로그램을 구성하는 서비스들을 지정
    + 각 서비스는 별개의 docker image 지정과 docker container 실행으로 구성됨
    + 각 서비스는 자신의 Dockerfile을 갖고 있어야함 아니면 docker hub등에서 이미지를 다운로드
    + 서비스 별로 포트번호, 환경변수, 디스크 볼륨 등을 지정해야함
    + 서비스 이름은 아무 이름이나 가능
- volumes과 networks는 앞서 사용된 것을 지정

## docker-compose VS docker compose
- v1: docker-compose
- v2: docker compose
- Docker 1.27부터 docker에 명령으로 compose가 추가됨
    + docker-compose 보다는 docker compose를 쓰는 것이 더 좋음
    + 별도로 docker-compose를 설치할 필요가 없음
        - 하지만 대부분의 문서가 docker-compose중심으로 만들어져 있다는 것은 감안해야 함

## docker-compose.yaml or docker-compose.yml
- docker compose 명령이 둘 중 하나를 찾음
    + 둘다 존재한다면 에러 발생
- 다른 이름의 파일을 사용하고 싶다면 -f 옵션 사용
    + docker compose -f docker-compose.test.yml up

## docker-compose.yml 작성 예
```bash

    frontend:
        build: ./frontend # dockerfile 경로

        ports:
            - 3001:3001

    vidly:
```

## docker-compose.yml 개선하기
### depends_on
- 서비스들간의 의존성이 있을 경우 먼저 실행되어야 하는 서비스들을 기술해야함
```bash
depends_on: 
    redis:
        condition: service_healthy

# short from
depends_on:
    - db
    - redis
# long form
depends_on
    db:
        condition: service_healthy # 서비스가 건강한지
    redis:
        condition: service_started # 서비스시작 건강상태는 관심없음
    worker:
        condition: service_completed_successfully # 성공적으로 끝난 다음에 실행(초기화)
```

### healthcheck
- 해당 서비스의 건강을 나타내주는 "체크"를 기술
- Dockerfile에서 기술가능한 기능이며 docker-compose에서 덮어쓰기 가능
```bash
healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost"]
    interval: 15s
    timeout: 5s
    retries: 3
    start_period: 10s
```

### volume
- volumes: db-data
    + db의 내용이 persistent해야함
```bash
db:
    volumes:
        - ./vote:/app

volumes:
    vote:
```

### network
- yml파일의 services들에 네트워크 지정해서 추가
- back-tier: redis, worker, db, vote, result
- front-tier: vote, result
```bash
vote:
    ...
    networks:
        - back-tier
        - front-tier

networks:
    - front-tier
    - back-tier
```

### db
- 환경 변수, volumes, healthcheck 추가 
```bash
environment:
    POSTGRES_USER: "postgres"
    POSTGRES_PASSWORD: "postgres"
```

### environment
- 해당 서비스가 컨테이너 안에서 실행될 때 환경변수들을 지정(Dockerfile의 ENV)
- MAP 문법 SHOW: "true"
- Array 문법 - SHOW=true

### redis
- volumes, healthcheck 추가

### worker
- depends_on, build 추가
- build를 통해 더 많은 빌드관련 정보를 넘겨줄 수 있음
```bash
build:
    context: ./myapp
    dockerfile: Dockerfile.dev
    args:
        ENVIRONMENT: development
```









## Airflow Docker docker-compose.yml 리뷰해보기
### docker-compose.yaml
- version
- x-airflow-common
    + airflow-common이라는 별칭 정의. 여러 서비스에서 공유하는 공통 구성을 정의
    + anchor라고 부르며 YML 파일 블록을 나중에 계승이란 형태로 재사용 가능하게 해줌
    + version, services, volumes, networks를 제외한 최상위 레벨 키워드는 모두 anchor
- services
    + postgres, redis
    + airflow
        - webserver
        - scheduler
        - worker
        - triggerer
        - init
- volumes
    + postgres-db-volume

### x-airflow-common
- &airflow-common 별칭 
    + image: ${AIRFLOW_IMAGE_NAME:-apache/airflow:2.5.1}
    + 같은 airflow 이미지가 모든 서비스 기본 이미지로 사용됨
- volumes:
    + host volumes, 모두 4개의 volume을 공유함
- redis
    + 모두 redis와 postgres가 정상작동할때까지 대기
- _PIP_ADDITIONAL_REQUIREMNETS: ${_PIP_ADDITIONAL_REQUIREMENTS:-}
    + Python과 관련된 필요한 모듈이 기술된 txt파일

- 별칭을 통해 x-airflow-common 내용 반복
- airflow init: condition: service_completed_successfully
